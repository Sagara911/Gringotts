# 美术素材管理器 — 设计思路

> 一个"懂画面、能用大白话搜、专为美术工作流打造、绝不锁定你数据"的本地素材库。
> 把 Eagle 的管理能力 + AI 语义搜索 + PureRef 的参考板体验合到一起。

---

## 一、产品定位

| 维度 | 说明 |
|------|------|
| **一句话定位** | 一个"懂画面、能用大白话搜、专为美术工作流打造"的本地素材库 |
| **对标产品** | Eagle（管理）、PureRef（参考板）、Billfish/MuseDAM/千鹿（平替） |
| **核心差异化** | AI 语义搜索 + 美术专用打标签 + 参考拼贴板 + **文件透明不锁定** |
| **目标用户** | 建模、原画、角色/场景美术等需要管理大量参考图/素材的人 |
| **目标** | 自用为主，长期可产品化发布 |

### 现状赛道分析
- **Eagle**（付费买断 $199）：素材管理一哥。短板：私有库格式、无移动端、搜索靠手打标签、不懂语义、英文支持差。
- **Billfish**（免费）：Eagle 平替，功能较弱。
- **MuseDAM**（网页）：协作好，但用户怕数据锁定、导不出。
- **千鹿**（免费）：多级标签、应用内素材管理。
- **Pixcall**：同步盘 + 素材搜索。
- **PureRef / VizRef**：参考板，不做管理。

> 结论：不做"又一个 Eagle"，要切它们没做好的点 —— **语义搜索 / 自动打标签 / 来源管理 / 参考板 / 数据不锁定**。

### 与 PureRef / Eagle 的定位区别（重要，别做偏）

| | PureRef | Eagle | **本程序** |
|---|---------|-------|------------|
| 本质 | 一块**无限画布**，把图摆好看 | 一个**素材库**，管理/搜索几万张图 | **素材库为主** + 内置一块画布 |
| 解决 | "把参考图铺开对着画" | "图怎么存、怎么找" | 两个都解决 |
| 核心交互 | 拖图、缩放、拼贴 | 文件夹/标签/搜索/网格 | 搜索浏览 + AI + 拼贴板 |

> **准确定位：主体是"会用 AI 的 Eagle"（导入/标签/搜索/管理），PureRef 式"无限画布拼贴板"是它内置的一个功能模块。**
> 比喻：PureRef 是"摊参考图的桌子"，本程序是"仓库 + 智能管理员"，想画时从仓库挑一批图一键摊到桌子上。
> ⚠️ 开发重心始终在 **管理 + 搜索 + AI**，参考板是顺手接入的好体验，别让重心偏到画布上。

## 二、市面工具短板（缺口地图）

> 来源：用户真实吐槽（Eagle 评价、Polycount、Blender Artists、知乎、少数派、3D 资产管理对比等）。

### 🔴 组织管理的硬伤
1. **标签系统太弱 / 没有层级** → 机会：**真正的层级标签 + 别名/同义词 + 多语言标签**。
2. **手动打标签没人愿意做**（所有工具的死结） → **AI 自动打标签，最大杀招**。
3. **库一大就"文件一团乱"** → 机会：**AI 自动聚类 + 去重 + 智能整理**。

### 🔴 AI 名不副实
4. **号称 AI 只会"认人脸认猫"** → 机会巨大：**语义搜图、反推提示词、画面分析**才是真 AI。

### 🔴 平台 / 同步 / 协作
5. **没有移动端 / iPad**（Eagle 最高频请求） → **只读手机查看端 + 同步**。
6. **多库管理、云同步弱**（依赖网盘凑合） → **原生云同步**。
7. **版本控制是灾难**（美术搞不定 Git） → **傻瓜版本管理**。

### 🔴 素材类型支持窄
8. **3D 模型/材质支持差** → 3D 缩略图预览 + 按贴图类型归类。
9. **视频/GIF/序列帧支持弱** → 动态素材预览。

### 🔴 版权与来源
10. **来源/作者信息丢失**（版权风险） → 自动记录来源 URL/作者。

### 🔴 数据主权（本轮调查新增、被低估的机会）
11. **怕私有格式锁死、数据导不出** → 国内用户强烈顾虑。机会：**文件原样存硬盘 + 元数据可一键导出，主打"绝不锁定用户"**。

### 🟡 杂项小痛点
- Eagle 只能 PayPal 付款，国内难付费；缺画面属性筛选；缺与 PS/Blender 联动。

### 📌 "既痛又适合你做"的优先级
| 缺口 | 痛度 | 难度 | 建议 |
|------|------|------|------|
| 真·AI 语义搜索/反推提示词 | 🔥🔥🔥 | 中 | **核心主打** |
| AI 自动打标签/整理 | 🔥🔥🔥 | 中 | **核心主打** |
| 文件透明 + 元数据可导出（不锁定） | 🔥🔥🔥 | 低 | **设计原则 + 宣传点，必做** |
| 来源/作者自动记录 | 🔥🔥 | 低 | **低成本高回报，必做** |
| 层级标签 + 多语言 + 批量操作 | 🔥🔥 | 低 | 早期就做 |
| 视觉近似去重 | 🔥🔥 | 中 | 复用 embedding，顺手做 |
| 移动端/平板查看 | 🔥🔥 | 高 | 路线图后期 |
| 3D 素材预览 | 🔥🔥 | 高 | 看是否主攻建模圈 |
| 傻瓜版本管理 | 🔥 | 中 | 差异化加分项 |

> **一句话**：市面工具最大共同短板是 **"管理靠手动 + AI 是假的 + 移动端/3D/来源 三缺 + 数据被锁定"**。

## 三、用户需求调查报告

> 调查范围：Capterra/ProductHunt 的 Eagle 评价、PureRef 论坛、Polycount、Blender Artists、知乎"Eagle 替代"系列、少数派素材管理横评、3D 贴图库讨论。样本：概念原画、3D/游戏美术、设计师（国内外）。

### 按"呼声强度"排序的核心需求

**🔥🔥🔥 最高频、最痛**
1. **自动化标签 / 不想手动打标** —— 提及最多。"每张图要打多个标签太麻烦""没批量打标就放弃了"。
2. **移动端 / 多设备访问** —— Eagle 评价里最频繁被提的限制。
3. **真正的云同步**（而非依赖网盘） —— 想要"原生同步工具"。

**🔥🔥 高频**
4. **怕被私有格式锁死 / 要文件透明** —— "Eagle 资料库是私有结构，不能随便动""怕网页方案导不出数据"。**国内用户强烈顾虑，被低估的机会**。
5. **层级标签 + 批量操作** —— 多级标签、批量重命名/打标。
6. **重复文件检测不彻底** —— Eagle 查重不是所有格式都管用。
7. **3D / 视频 / 特殊格式支持差** —— 有工作室自己写工具来排序，说明现成方案满足不了。

**🔥 中频 / 加分**
8. 来源/出处丢失（版权隐患）。 9. UI 不够顺手。 10. 价格/英文支持/国内付费门槛。 11. "为项目囤积→下次用不上"，库膨胀冗余。

### 调查带来的新洞察（已并入设计）
1. **"不锁定用户"是差异化卖点** —— 文件透明存放 + 一键导出元数据，写进宣传点。
2. **去重要"跨格式 + 视觉近似"** —— 有 embedding 正好能做"长得像也查得出"。
3. **"跨项目复用"是更深痛点** —— 按内容/风格找而非按项目文件夹找，正是语义搜索价值。
4. **已有人自己造工具** —— 需求真实、现成方案不够、做出来有人用。

## 四、整体架构（分层）

```
┌─────────────────────────────────────────────┐
│  UI 层（前端 React）                            │
│  浏览/网格 · 详情 · 搜索框 · 标签 · 参考板        │
├─────────────────────────────────────────────┤
│  应用逻辑层（前端 + Tauri 命令桥接）              │
│  导入流程 · 搜索调度 · 标签管理 · 拼贴生成         │
├─────────────────────────────────────────────┤
│  核心服务层（Rust 后端）                         │
│  ① 文件扫描   ② 缩略图生成   ③ 元数据管理         │
│  ④ AI 推理调度   ⑤ 向量检索                      │
├─────────────────────────────────────────────┤
│  数据层                                        │
│  SQLite（元数据/标签） + sqlite-vec（向量）       │
│  + 本地缩略图缓存目录（原图原样留硬盘，不动）        │
├─────────────────────────────────────────────┤
│  AI 层（可插拔）                                │
│  本地模型(默认)  ⇄  云端 API(可选开关)            │
└─────────────────────────────────────────────┘
```

**关键原则：**
- **AI 层可插拔**：上层只调统一接口（`embed()` / `vision()`），换模型不动主程序。
- **数据不锁定**：原图原样留在用户硬盘，元数据独立存储且可一键导出。

## 五、功能模块（按依赖顺序）

| # | 模块 | 内容 | 依赖 AI? |
|---|------|------|---------|
| 1 | **资产导入** | 扫描文件夹、拖入、去重检测、读取 EXIF/来源 | 否 |
| 2 | **缩略图引擎** | 生成多尺寸缩略图、缓存、懒加载 | 否 |
| 3 | **元数据库** | 图片信息、文件夹、标签、来源、收藏 | 否 |
| 4 | **浏览 UI** | 网格虚拟滚动、详情页、基础筛选 | 否 |
| 5 | **颜色分析** | 提取主色调/配色，按颜色筛选 | 否（纯算法） |
| 6 | **语义搜索** | 文字搜图、以图搜图（向量检索） | ✅ 嵌入模型 |
| 7 | **自动打标签** | 导入时自动生成标签/风格分类 | ✅ 视觉模型 |
| 8 | **参考拼贴板** | 选图生成 PureRef 式画布 | 否 |
| 9 | **采集插件** | 浏览器一键存图带来源/作者 | 否 |

> 模块 **1~5 + 8** 不依赖 AI，是可独立自用的骨架。**6、7** 是 AI 增强层。

### 模块⑧详解：内置参考板（PureRef 模块）

把 PureRef 的"无限画布参考板"作为内置模块，和素材库无缝打通。

**对齐 PureRef 的基本盘：**
- 无限画布，自由拖放、缩放、旋转
- 多选对齐、网格吸附
- 加文字注释、画框/箭头标注（PureRef 2.x 才补，这里一步到位）
- 画布存成工程文件，下次接着用

**比 PureRef 强（因为背后有库）：**
- **直接从库里拖图进画布**，不用翻文件夹重新导入
- 画布上的图可**反查**："这张在库里还有哪些相似的" → 一键补更多参考
- 选中图直接调 AI：反推提示词、分析配色
- 一个素材项目 ↔ 一块画布，自动关联

**实现方式（不从零造）：**
- 用开源无限画布库 **tldraw**（推荐）或 Excalidraw，自带拖拽/缩放/撤销/画框，主要写"自定义图片卡片 + 从库导入"的桥接，省约 80% 工作量。
- 在 Tauri 里作为一个独立画布视图，与主库共用同一份数据。

> **排期**：放在阶段二（"找到图之后"的环节），不进 MVP。

### 关于「素材处理」：交给 Dobby（独立网页工具）

本程序**不自己做处理（抠图/压缩/贴图/超分等）**，这部分由作者已有的纯前端工具站 **Dobby**（`D:\Game\toolkit`，线上 https://dobby-aih.pages.dev/ ）承担。

- **分工**：本程序 = 管理/搜索/AI/参考板（"库"）；Dobby = 处理（"工坊"）。一套流水线的互补两半。
- **Dobby 现有能力**：AI 抠图、去水印、压缩/转格式、调色、法线贴图、贴图生成、通道检查、图集打包/拆帧、GIF、视频处理/超分、OCR、EXIF、批量重命名等（22~26 款，纯前端、不上传、PWA）。
- **现阶段策略**：Dobby 保持网页独立持续更新，本程序**不重复造处理功能**。
- **联动（先轻量）**：库内右键图片 →「用 Dobby 处理」→ 打开对应 Dobby 工具页（每个工具有独立 URL，如 `tools/ai-cutout.html`）。
- **未来（若网页处理不够用）**：再考虑把高频工具（抠图/贴图/超分）以 WebView 内嵌或移植为本地模块，处理结果自动回存为新版本。

> 决定（2026-06）：**暂不合并，留轻量联动接口；以后视情况再内置。**

## 六、AI 层设计（核心）

### ⚠️ 关键区分：两类 AI 不是一回事

| 功能 | 需要的模型类型 | 对话模型（Gemma/DeepSeek）能做吗？ |
|------|--------------|-------------------------------|
| **搜图**（文字/以图搜图） | **嵌入模型 Embedding**（输出向量） | ❌ 不能，对话模型不输出向量 |
| **打标签/描述/整理** | **视觉对话模型 VLM**（看图输出文字） | ✅ 正是强项 |

### AI 层拆成两个 Provider
```
AI 层
├── ① Embedding Provider（管"搜图"）
│     默认：本地 SigLIP / Chinese-CLIP（离线免费）
│     可选：Jina / Cohere multimodal embedding API
│     ⚠️ 不轻易开放更换（换了要重建索引）
└── ② Vision Provider（管"打标签/描述/整理"）
      默认：本地 Gemma 4（2B/4B 轻量，12B 高质量）
      可选：用户自填 OpenAI 兼容 API → DeepSeek V4 Vision / GPT / Qwen-VL
```

### Vision Provider 可插拔配置（用户可填）
```
Provider:     [本地 Gemma4 ▾] / OpenAI / DeepSeek / 自定义
API Base URL: https://api.deepseek.com
API Key:      sk-xxxx
Model:        deepseek-v4-vision
```
> 遵循 **OpenAI 兼容格式**（DeepSeek、Qwen、Ollama 都兼容），一套代码接所有家。

### AI 选型定稿
| 能力 | 本地默认 | 可选云端 / 补充 |
|------|---------|----------------|
| **搜图（embedding）** | SigLIP / Chinese-CLIP（离线免费） | Jina / Cohere multimodal embedding API |
| **打标签/理解（VLM）** | Gemma 4（2B/4B 默认，12B 可选，Apache 2.0 可商用） | DeepSeek V4 Vision / GPT / Qwen-VL（用户自填 key） |
| **美术专用打标签** | WD14 Tagger（插画/二次元识别更准） | — |

## 七、AI 扩展能力（VLM 解锁的玩法）

> ⚠️ **这些是路线图能力，不是 MVP**。先把"扫描→缩略图→浏览→搜图"做扎实。

### 🎨 最贴美术工作流（优先）
1. **从参考图反推 AI 绘画提示词** ⭐ —— 拖图自动生成 SD/MJ 提示词。
2. **用大白话跟素材库对话** —— "把夜景、冷色调、带人物的挑出来"。
3. **自动写描述 / 自动命名**。
4. **画面分析 / 拉片** —— 打光、构图、配色讲解。

### 🧩 锦上添花
5. 智能聚类/整理。 6. 图里文字 OCR。 7. 风格自动归类。 8. 作品点评。

### ⚙️ 扩展架构：能力 + Prompt 模板
- 底层一个 `vision(image, prompt)` 接口，每个功能本质是**换一段 prompt 模板**。
- **加新功能 = 加 prompt 模板 + 入口按钮**，不动核心代码；进一步可做"指令库/插件"。

### ⚠️ 两条底线
1. **MVP 不碰这些**。 2. **别跑偏成"什么都能干的 AI 工具"**，核心永远是"美术素材"。

## 八、模型更新策略（让升级无痛）

### 🟢 视觉对话模型（Gemma 类）—— 更新极简单
1. **权重按需下载 + 清单管理**（不打包进安装包）：
   ```json
   // models.json —— 更新模型只要改清单，不动代码
   {
     "gemma-4-4b":  { "url": "...", "size": "2.5GB", "default": true },
     "gemma-4-12b": { "url": "...", "size": "7GB" },
     "gemma-5-xxx": { "url": "...", "size": "..." }   // 出新版加一行
   }
   ```
2. **推理运行时抽象层**：用 Ollama / llama.cpp（GGUF），社区出新版量化后只需加链接，代码不改。
3. 走**标准格式（GGUF/ONNX）+ 通用运行时**。

### 🔴 嵌入模型（搜图用）—— 换版要小心
> **换嵌入模型 = 全库向量作废 = 必须重跑所有图。**
- 数据库给每条向量记 `model_version`，升级时检测不一致 → 后台重建索引。
- 嵌入模型尽量一开始选定；对话模型可随意升级。

### 8.x AI 运行时与分发（已实现 + 待办记录）

**已实现（开发阶段）**
- Vision Provider 走本地 **Ollama**（OpenAI 兼容接口 `http://localhost:11434/v1`）。
- 默认模型 **`gemma4:12b`**（7.6GB，适配 12GB 显存全程 GPU；可按硬件换 e4b/26b/31b）。
- 可用环境变量覆盖以接云端：`GRINGOTTS_AI_BASE` / `GRINGOTTS_AI_MODEL` / `GRINGOTTS_AI_KEY`（如 DeepSeek V4 Vision / GPT）。
- 踩坑：早期 `gemma4:latest` 的视觉投影/模板损坏（图片变 `?`），改用官方 `gemma4:e4b`/`12b` 修复。

**待办：发给"没装 Ollama"的用户时（打包分发阶段再做，非当前）**
- **权重不进安装包**（否则 8~20GB）；改为**首次运行下载模型 + 进度条**。
- 运行时二选一：
  - **路线 A（推荐/务实）**：程序自带或自动下载 **Ollama 运行时**，作为后台子进程托管，用户无感；保留 OpenAI 兼容架构。
  - **路线 B（彻底自包含）**：Rust 绑定 **内嵌 llama.cpp**，自己加载 GGUF，不依赖 Ollama；最独立但工程量大（GPU 后端、视觉投影）。
- 决定（2026-06）：当前用现成 Ollama；分发方案待功能稳定后再做。

## 九、需求难度评估（这些需求好加吗）

> 基于本架构（Tauri + SQLite + sqlite-vec + 可插拔 AI 层）评估。

| 需求 | 难度 | 说明 |
|------|------|------|
| 文件透明存放 + 元数据可导出 | 🟢 易 | 设计决策：文件留硬盘，元数据存 SQLite + 导出 JSON/CSV，几乎零额外成本 |
| 层级标签 + 批量打标/重命名 | 🟢 易 | 标准数据库 + UI |
| 来源/作者记录（导入侧） | 🟢 易 | 读图片元数据、记录拖入 URL |
| 反推提示词 / 画面分析 | 🟢 易 | 给 VLM 换 prompt 模板，几乎免费 |
| AI 自动打标签 | 🟡 中 | 架构已留 Vision Provider，难在调 prompt 和准确率 |
| 语义搜索（文字/以图搜图） | 🟡 中 | 接 embedding + sqlite-vec，要调优 |
| 视觉近似去重 | 🟡 中 | 复用 embedding 算相似度 |
| 视频/GIF 缩略图 | 🟡 中 | 接 ffmpeg 抽帧 |
| 浏览器采集插件 | 🟡 中 | 单独写扩展 + 通信 |
| 3D 模型/材质缩略图 | 🔴 难 | 要离屏渲染，坑多 |
| 真·云同步 | 🔴 难 | 后端基建 + 冲突合并，另一个量级 |
| 移动端（哪怕只读） | 🔴 难 | 等于再做一个 App + 依赖云同步 |

**结论**：
- 一大半是"易/中"，**高回报项（文件透明、层级标签、来源、反推提示词）几乎不难**。
- 真正难的只有 3 个：**3D 预览、云同步、移动端**——均属阶段三，可先不碰。
- **AI 层搭好后，"自动标签/语义搜索/反推提示词/视觉去重"是同一套底座长出来的，边际成本很低。**

### B. 需求结论（按目标分层）

| 产品 | 形态/价格 | 强项 | 软肋（你的机会） |
|------|----------|------|----------------|

### D. 成败关键 = 差异化三张牌
| 差异化牌 | 打击的软肋 | 强度 |
|---------|-----------|------|

## 十一、技术栈定稿

| 层 | 选型 |
|---|------|
| 壳 | **Tauri 2.0**（Rust，扫海量本地图、内存占用低、安装包小） |
| 前端 | React + TS + 虚拟滚动（react-virtuoso） |
| 元数据 | SQLite |
| 向量检索 | **sqlite-vec**（轻、与 SQLite 一体） |
| 本地 AI 运行时 | Ollama / llama.cpp（GGUF）；嵌入用 candle / onnxruntime |
| 缩略图 | Rust 端 image crate 生成缓存 |

**为什么桌面端而非 Web**：要管本地硬盘成千上万张图，Web 端无法很好访问本地大文件夹。Tauri 比 Electron 省内存、安装包小、跨平台打包方便。

## 十二、分阶段路线图

### 阶段一：自己先用起来（MVP，核心"管得住"）
- 扫描本地文件夹入库（文件原样留硬盘）、缩略图网格（几万张不卡）
- 文件夹 + 多级标签双重组织、批量操作、看大图、基本元信息
- 文件名/标签搜索、主色调提取 + 按颜色筛选
- 元数据可导出（不锁定）
- → 到这里即可日常替代 Eagle

### 阶段二：差异化（核心"搜得到"）
- 本地 SigLIP/CLIP 向量化 → 文字搜图 + 以图搜图
- 视觉近似去重、来源/作者自动记录
- 可选云端 API 开关（Vision Provider）
- 一键生成 PureRef 式参考拼贴板

### 阶段三：产品化（核心"卖得出"）
- 浏览器插件一键采集（带来源/作者）
- AI 扩展能力：反推提示词、画面分析、智能整理（见第七章）
- 风格自动分类、视频/3D 预览
- 云同步 / 团队共享 / 移动端只读查看（订阅点）
